# MCP Usage

AOI Package 2 exposes the Package 1 core engine through read-only MCP tool functions. If the Python MCP SDK is installed, `ai_objective_index.mcp_server` can register a simple MCP server. If the SDK is missing, local tool functions and manifest generation still work.

## Tool Set

Tools are described in `schemas/mcp_tools.schema.json` and generated by `python -m ai_objective_index.mcp_manifest`:

- `search`
- `fetch`
- `search_objectives`
- `rank_options`
- `compare_tools`
- `explain_score`
- `get_source_trace`
- `list_missing_fields`
- `generate_decision_receipt`

## Agent-Native Discovery Pack

AOI-AGENT-ADOPTION-2 wires agent-facing discovery and preflight MCP tools for the unpublished 0.3.0a2 package candidate:

- `get_aoi_capability_card`
- `discover_capabilities_for_objective`
- `preflight_capability_for_use`
- `explain_aoi_agent_use`
- `list_aoi_agent_examples`

These tools are read-only local helpers. Discover mode returns useful source-traced candidates even when candidates remain HOLD. Preflight mode separates `tool_available` from `tool_authorized`, preserves missing fields, and blocks external action, certification, and product-readiness requests. The tools do not call live MCP servers, execute tools, call external APIs, upload packages, publish registry metadata, handle tokens, certify security, prove correctness, guarantee quality, claim product readiness, or authorize external action.

## Read-Only Contract

All v0.1 tools are read-only. They may search, rank, compare, explain, retrieve traces, list missing fields, and generate decision receipts from existing data. They must not perform payment, booking, login, email sending, form submission, purchase, or contract signing.

They also do not crawl URLs, fetch network data, connect accounts, claim suppliers, verify suppliers, modify profiles, or implement a REST API.

## Package 9C Objective Router Tools

Package 9C adds read-only vNext MCP helpers:

- `route_objective(query, objective, domain="mcp_servers", data_scope="sample", limit=10, constraints={})`
- `get_capability_trust(capability_id, data_scope="sample")`
- `explain_route_decision(capability_id, objective=None, data_scope="sample")`

These tools return route decisions, evidence summaries, risk boundaries, missing fields, and must-not-claim terms. They do not fetch network data, run probes, execute external tools, or perform payment, booking, login, email, purchase, form submission, or contract actions.

## Package 9D ExecutionReceipt Tools

Package 9D adds local receipt-memory MCP helpers:

- `submit_execution_receipt(receipt)`
- `get_execution_receipt(receipt_id)`
- `list_capability_receipts(capability_id, limit=20)`
- `get_capability_receipt_memory(capability_id)`
- `route_objective_with_receipts(query, objective, domain="mcp_servers", data_scope="sample", limit=10, constraints={})`

`submit_execution_receipt` writes only to the local receipt store. It is not an external action and does not contact services. Receipt memory can add warnings or downgrade routes; it cannot verify capabilities, certify security, guarantee quality, or authorize payment, booking, login, email, purchase, form submission, or contract actions.

## Package 6C data_scope

Package 6C adds optional `data_scope` support to MCP helper functions while preserving `sample` as the default.

Allowed values:

- `sample`
- `generated`
- `integrated`

Example:

```json
{
  "tool": "search_objectives",
  "arguments": {
    "query": "OCR API with free tier and rate limits",
    "data_scope": "integrated",
    "limit": 5
  }
}
```

Generated-scope objects remain `EXTRACTED_UNVERIFIED`. MCP output is still read-only and does not crawl, buy, book, log in, send email, submit forms, claim suppliers, verify suppliers, or sign contracts.

When the MCP SDK is missing, the server command prints:

```text
MCP SDK not installed. Tool functions and manifest are available.
```

Package 2 tests tool functions locally without requiring a live MCP client.

## Package 6D-S Source-Governed Output

MCP tool outputs may include:

- `claim_ceiling`: the highest safe claim tier;
- `not_asserted`: claims AOI explicitly does not make;
- `obstructions`: HOLD/BLOCK certificates with next actions;
- `use_rights`: read/rank/compare/quote allowed, train/share/memory/action held or blocked;
- `action_boundary`: allowed read-only actions and blocked external actions.

MCP recommendations are not action licenses. The tool layer remains read-only and local-data-only.

## Package 6D data_scope

MCP helpers accept `data_scope` where relevant:

- `search_objectives(..., data_scope="sample|generated|integrated|curated|public_beta|mcp_registry|public_beta_mcp")`
- `rank_options(..., data_scope="sample|generated|integrated|curated|public_beta|mcp_registry|public_beta_mcp")`
- `compare_tools(..., data_scope="sample|generated|integrated|curated|public_beta|mcp_registry|public_beta_mcp")`
- `explain_score(..., data_scope="sample|generated|integrated|curated|public_beta|mcp_registry|public_beta_mcp")`
- `get_source_trace(..., data_scope="sample|generated|integrated|curated|public_beta|mcp_registry|public_beta_mcp")`
- `list_missing_fields(..., data_scope="sample|generated|integrated|curated|public_beta|mcp_registry|public_beta_mcp")`
- `generate_decision_receipt(..., data_scope="sample|generated|integrated|curated|public_beta|mcp_registry|public_beta_mcp")`

Default remains `sample`. Generated and curated records remain `EXTRACTED_UNVERIFIED`.

Package 7C adds:

- `curated`: manually curated candidates only;
- `public_beta`: curated candidates that pass the evidence gate.
- `mcp_registry`: local Official MCP Registry intake records;
- `public_beta_mcp`: registry records that pass the calibrated MCP Registry candidate gate.

If `public_beta` or `public_beta_mcp` has no ready objects, MCP search returns a warning instead of silently falling back to fake sample data. `public_beta_mcp` rows are registry metadata candidates only: not verified, not security certified, not quality guaranteed, not purchasing advice, and not action permission.

## Package 7A Generic search/fetch

Package 7A adds generic MCP compatibility wrappers:

- `search(query, data_scope="sample", limit=10)`: returns objective-ranked candidates using AOI search logic.
- `fetch(object_id, data_scope="sample")`: returns a full local object record with score, source traces, missing fields, limitations, and forbidden actions.

These wrappers are read-only. They do not crawl, fetch remote URLs, buy, book, pay, log in, email, submit forms, purchase, connect accounts, verify suppliers, or sign contracts.

## Example Flow

1. Agent calls `search_objectives` with an objective request.
2. Agent calls `rank_options` to obtain objective scores.
3. Agent calls `get_source_trace` for evidence behind important fields.
4. Agent calls `generate_decision_receipt` to record why an option was selected.

The receipt is an explanation artifact, not a purchase order, contract, or approval.

## Package 9E Probe-before-Use MCP Tools

Package 9E adds local-only probe tools:

- `plan_probe_before_use`
- `run_local_probe_plan`
- `get_probe_receipt`
- `get_capability_probe_memory`
- `route_objective_with_probes`

These tools inspect local metadata and source-trace records only. They do not call live MCP servers, execute external tools, fetch URLs, run a gateway, certify security, guarantee quality, verify capabilities, or authorize payment, booking, login, email, purchase, or contract actions.
