# ROE Public/Private Split

ROE public surfaces can show enough structure for developers and AI agents to integrate with the products without exposing the private kernel.

| Public Surface | Allowed |
| --- | --- |
| Schemas | Packet, receipt, residual, and route-decision shapes |
| API/MCP shapes | Read-only or local-memory interfaces |
| Docs | High-level methodology and claim boundaries |
| Examples | Small sample fixtures and conservative demos |
| Decisions | ALLOW/HOLD/BLOCK labels and explanations |

| Private Surface | Reserved |
| --- | --- |
| Calibration | ranking-weight values and threshold policy |
| Defense | anti-gaming rules and private negative-control banks |
| Trust priors | provider priors and commercial routing policy |
| Probe bank | private probe seeds and validator portfolios |
| Data strategy | private acquisition, freshness, and enterprise policies |

This split supports open adoption while preserving the harder-to-copy operational kernel. Public examples must remain conservative and must not imply verification, security certification, quality guarantee, legal sufficiency, product readiness, or action authorization.
